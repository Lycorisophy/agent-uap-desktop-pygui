# 智能体记忆系统设计方案参考指南（完整版）

## 1. 系统概述

本系统旨在为智能体项目空间构建长期记忆能力，包含两大核心流程：

1. **后台定时知识抽取**：从项目空间的历史对话、文档中自动抽取事件、事件关系、事实三类知识，并与已上传的知识库内容统一存入向量数据库，同时记录抽取进度防止重复。
2. **统一RAG检索工具**：将记忆检索封装为单一工具，由大模型自主判断是否调用、调用参数，内部支持混合检索与可选重排序，返回高质量记忆片段。

```
└─────────────────┘      └─────────────────┘      └────────┬────────┘
                                                            │
                                                            ▼
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐

```

## 2. 数据模型设计

### 2.1 原始信息单元 Episode

在进入向量库之前，所有历史消息、文档片段首先以 Episode 形式记录，作为抽取的源头和进度锚点。

```python
class Episode(BaseModel):
    id: str                      # UUID
    project_space_id: str        # 所属项目空间
    source_type: str             # 如 "chat", "issue", "doc_chunk"
    content: str                 # 原始文本内容
    created_at: datetime         # 发生时间戳
    processed: bool = False      # 是否已完成抽取
    processed_at: Optional[datetime] = None
```

Episode 可存储在 SQLite 或 PostgreSQL 中，用于断点续传。

### 2.2 四种知识类型 Schema

统一存入 Milvus 的知识对象结构如下（仅展示核心字段，实际存储时会扁平化为 Milvus 字段）：

```python
class KnowledgeType(str, Enum):
    EVENT = "event"
    EVENT_RELATION = "event_relation"
    FACT = "fact"
    KNOWLEDGE_BASE = "knowledge_base"

class BaseKnowledge(BaseModel):
    id: str                                 # 知识唯一ID
    project_space_id: str
    knowledge_type: KnowledgeType
    content: str                            # 用于向量化和展示的文本
    source_episode_ids: List[str]           # 溯源
    created_at: datetime
    valid_from: datetime                    # 双时序：事实生效时间
    valid_to: Optional[datetime] = None     # 双时序：事实失效时间
    confidence: float                       # 抽取置信度
    metadata: Dict[str, Any]                # 扩展元数据
```

**具体类型的特殊字段**（统一放入 `metadata` JSON 字段）：

- **Event**：`subject`, `predicate`, `object`, `timestamp`, `location`
- **EventRelation**：`source_event_id`, `target_event_id`, `relation_type`
- **Fact**：`subject`, `attribute`, `value`
- **KnowledgeBase**：`doc_id`, `chunk_index`, `title`

## 3. 存储架构设计

### 3.1 存储分离策略

| 组件 | 存储内容 | 作用 |
|------|----------|------|
| **Milvus** | 向量（稠密+稀疏）、基础元数据（`id`、`project_space_id`、`knowledge_type`、`created_at`、`confidence`、`metadata`） | 高效向量检索与标量过滤 |
| **SQLite / PostgreSQL** | 知识的 `content` 字段全文、Episode 原文 | 原文回溯，节省 Milvus 存储成本 |

检索时，先从 Milvus 获取 Top-K 的 `id` 列表，再批量从 SQLite 查询 `content` 内容。

### 3.2 Milvus Collection Schema

```python
from pymilvus import Collection, CollectionSchema, FieldSchema, DataType

fields = [
    FieldSchema(name="id", dtype=DataType.VARCHAR, is_primary=True, max_length=100),
    FieldSchema(name="project_space_id", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="knowledge_type", dtype=DataType.VARCHAR, max_length=32),
    FieldSchema(name="dense_vector", dtype=DataType.FLOAT_VECTOR, dim=1024),  # 根据模型调整
    FieldSchema(name="sparse_vector", dtype=DataType.SPARSE_FLOAT_VECTOR),    # Milvus 2.4+ 支持
    FieldSchema(name="created_at", dtype=DataType.INT64),
    FieldSchema(name="confidence", dtype=DataType.FLOAT),
    FieldSchema(name="metadata", dtype=DataType.JSON),                         # Milvus 2.4+ 支持 JSON
]

schema = CollectionSchema(fields, description="智能体记忆集合")
collection = Collection(name="agent_memory", schema=schema)
```

### 3.3 索引配置

```python
# 1. 稠密向量索引（语义检索）
dense_index_params = {
    "index_type": "HNSW",
    "metric_type": "COSINE",
    "params": {"M": 16, "efConstruction": 200}
}
collection.create_index("dense_vector", dense_index_params)

# 2. 稀疏向量索引（BM25 关键词检索）
sparse_index_params = {
    "index_type": "SPARSE_INVERTED_INDEX",
    "metric_type": "IP"   # 内积，用于 BM25 得分
}
collection.create_index("sparse_vector", sparse_index_params)

# 3. 标量字段索引（加速过滤）
# 对 project_space_id, knowledge_type 创建倒排索引
collection.create_index("project_space_id", {"index_type": "INVERTED"})
collection.create_index("knowledge_type", {"index_type": "INVERTED"})
collection.create_index("created_at", {"index_type": "INVERTED"})
# 对 JSON 内字段的索引（可选）
collection.create_index("metadata['confidence']", {"index_type": "INVERTED"})
```

### 3.4 启用 BM25 自动生成

在数据插入时，Milvus 会根据配置对 `content` 字段自动生成稀疏向量。需要在 Schema 中声明该字段与 BM25 分析器的关联（Milvus 2.5+）：

```python
# 在创建 collection 时通过 properties 配置 BM25
collection_properties = {
    "mmap.enabled": True,
    "sparse.vector.index": {
        "bm25": {
            "field": "content",      # 指定用于生成 BM25 稀疏向量的文本字段
            "analyzer": "english"    # 或 "chinese"
        }
    }
}
```

数据插入时只需提供 `content` 文本，Milvus 会自动计算并填充 `sparse_vector` 字段。

## 4. 后台定时知识抽取

### 4.1 抽取流程概览

```python
async def scheduled_extraction(project_space_id: str):
    # 1. 获取分布式锁，避免并发
    lock = await acquire_lock(f"extract:{project_space_id}")
    if not lock:
        return
    
    try:
        # 2. 查询未处理的 Episode
        episodes = await db.query(
            "SELECT * FROM episodes WHERE project_space_id=? AND processed=false",
            project_space_id
        )
        
        for ep in episodes:
            # 3. 调用大模型抽取事件、关系、事实
            events = await extract_events(ep.content)
            relations = await extract_relations(events, ep)
            facts = await extract_facts(ep.content)
            
            # 4. 生成 Milvus 插入数据（包含向量化）
            records = prepare_milvus_records(events, relations, facts, project_space_id)
            await milvus_insert(records)
            
            # 5. 将原文内容存入 SQLite
            await sqlite_insert([r.id for r in records], [r.content for r in records])
            
            # 6. 标记 Episode 为已处理
            await db.execute(
                "UPDATE episodes SET processed=true, processed_at=? WHERE id=?",
                datetime.now(), ep.id
            )
        
        # 7. 处理知识库文件（若有新上传）
        await process_knowledge_base_files(project_space_id)
        
        # 8. 更新整体进度表
        await update_progress(project_space_id, last_processed_id=episodes[-1].id)
    
    finally:
        await release_lock(lock)
```

### 4.2 断点续传与进度记录

**进度表结构**（存于 SQLite/PostgreSQL）：

```sql
CREATE TABLE extraction_progress (
    project_space_id VARCHAR(64) PRIMARY KEY,
    last_processed_episode_id VARCHAR(100),
    last_processed_at TIMESTAMP,
    total_episodes_processed INT DEFAULT 0,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

每次抽取完成后更新 `last_processed_episode_id`。下次任务启动时，查询 `id > last_processed_episode_id` 的未处理 Episode，实现严格的增量处理。

若需支持不同知识类型的独立进度，可在表中增加 `knowledge_type` 字段并建立复合主键。

### 4.3 抽取提示词设计要点

抽取模块需要调用大模型进行结构化信息提取，建议：

- **事件抽取**：要求输出 JSON 格式，包含 `subject`, `predicate`, `object`, `timestamp`, `confidence`。
- **事件关系抽取**：基于已抽取的事件列表，识别 `causes`, `precedes`, `contradicts` 等关系。
- **事实抽取**：提取对话中确认的、相对稳定的断言（如“用户名叫张三”、“项目截止日期是周五”）。

可以引入 Graphiti 或 Ryumem 等开源库中的提示词模板，或者自行设计 few-shot 示例以保证输出格式稳定。

## 5. 统一 RAG 检索工具

### 5.1 工具定义

工具以 LangChain Tool 形式封装，大模型可通过 function calling 自主调用。

```python
from langchain.tools import BaseTool
from pydantic import BaseModel, Field
from typing import List, Optional

class MemorySearchInput(BaseModel):
    query: str = Field(description="检索查询语句")
    knowledge_types: List[str] = Field(
        default=["event", "event_relation", "fact", "knowledge_base"],
        description="检索的知识类型"
    )
    top_k: int = Field(default=5, description="最终返回结果数量")
    use_hybrid: bool = Field(default=True, description="启用混合检索（BM25+向量）")
    use_rerank: bool = Field(default=False, description="启用Reranker精排")
    recall_top_n: int = Field(default=50, description="重排序前的召回数量")
    time_range_days: Optional[int] = Field(None, description="时间范围（天）")
    min_confidence: float = Field(0.6, description="最低置信度阈值")

class UnifiedRAGTool(BaseTool):
    name = "memory_search"
    description = """从智能体记忆中检索信息。可用于查询历史事件、关系、事实或知识库内容。"""
    args_schema = MemorySearchInput
    
    def _run(self, **kwargs) -> str:
        # 实现检索逻辑
        return self._execute_search(**kwargs)
```

### 5.2 混合检索与重排序实现

```python
def _execute_search(self, query: str, knowledge_types: List[str], top_k: int,
                    use_hybrid: bool, use_rerank: bool, recall_top_n: int, ...):
    # 1. 生成稠密向量
    dense_emb = self.embed_model.encode(query)
    
    # 2. 构建过滤表达式
    filter_expr = f"project_space_id == '{self.project_id}'"
    if knowledge_types:
        filter_expr += f" && knowledge_type in {knowledge_types}"
    if time_range_days:
        since_ts = int((datetime.now() - timedelta(days=time_range_days)).timestamp())
        filter_expr += f" && created_at >= {since_ts}"
    if min_confidence:
        filter_expr += f" && confidence >= {min_confidence}"
    
    # 3. 多路召回
    if use_hybrid:
        # 混合检索：同时使用稠密向量和稀疏向量
        search_params = {
            "metric_type": "COSINE",
            "params": {"nprobe": 16}
        }
        results = self.collection.hybrid_search(
            reqs=[
                {"vector": dense_emb, "anns_field": "dense_vector", "param": search_params, "expr": filter_expr, "limit": recall_top_n},
                {"vector": None, "anns_field": "sparse_vector", "param": {"metric_type": "IP"}, "expr": filter_expr, "limit": recall_top_n}
            ],
            rerank=WeightedRanker(0.7, 0.3),  # 权重融合
            limit=recall_top_n,
            output_fields=["id", "knowledge_type", "created_at", "confidence", "metadata"]
        )
    else:
        # 仅语义检索
        results = self.collection.search(
            data=[dense_emb],
            anns_field="dense_vector",
            param=search_params,
            expr=filter_expr,
            limit=recall_top_n,
            output_fields=[...]
        )
        results = results[0]  # 单查询
    
    # 4. 可选：Reranker 精排
    if use_rerank:
        # 从 SQLite 批量获取 content
        ids = [hit.id for hit in results]
        contents = sqlite.fetch_contents(ids)
        # 计算精排得分
        rerank_scores = self.reranker.compute_scores(query, contents)
        # 重排序
        for hit, score in zip(results, rerank_scores):
            hit.score = score
        results.sort(key=lambda x: x.score, reverse=True)
    
    # 5. 截取 top_k 并组装返回
    final_ids = [hit.id for hit in results[:top_k]]
    final_contents = sqlite.fetch_contents(final_ids)
    return self._format_response(query, final_ids, final_contents)
```

### 5.3 大模型自主决策集成

在 Agent 循环中，将工具绑定至 LLM：

```python
from langgraph.prebuilt import create_react_agent

tools = [UnifiedRAGTool()]
llm = ChatOpenAI(model="gpt-4").bind_tools(tools)

agent_executor = create_react_agent(llm, tools)
```

当用户提问时，模型自行判断是否需要调用 `memory_search`，并自动生成包含 `query`、`knowledge_types` 等参数的调用请求。

## 6. 重排序（Rerank）选型与部署建议

| 场景 | 推荐方案 | 说明 |
|------|----------|------|
| **追求极致精度** | `Qwen3-Reranker` 或 `Cohere Rerank API` | 企业级高精度，Cohere 省运维成本 |
| **中英文混合、本地部署** | `bge-reranker-v2-m3` | 开源社区活跃，性能优秀 |
| **低延迟、资源受限** | `ms-marco-MiniLM-L6-v2` | 轻量级，CPU 可运行 |
| **无需额外部署** | 仅依赖 Milvus 的 `WeightedRanker` 融合 | 将 `use_rerank=False` 即可 |

在代码实现中，建议将 Reranker 封装为可插拔的接口，方便切换不同模型。

## 7. 工程落地注意事项

1. **幂等性**：基于 Episode 的 `processed` 标志位保证每条消息只抽取一次。若抽取过程异常中断，重启任务后会重新处理未标记完成的 Episode，不会重复写入（因为知识 ID 可基于内容哈希生成）。
2. **批量写入优化**：Milvus 插入推荐每 1000 条一批，Embedding 生成也可批量调用 API。
3. **监控与告警**：记录每次抽取任务的耗时、处理条数、失败记录；设置告警阈值（如连续 3 次失败）。
4. **版本管理**：当抽取提示词或模型升级时，可增加 `extractor_version` 字段，支持对历史 Episode 进行重抽取。
