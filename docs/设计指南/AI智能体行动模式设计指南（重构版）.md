## 1. 概述：AI智能体与行动模式体系

### 1.1 什么是AI智能体

AI智能体（Agent）是一种能够感知环境、进行推理、制定计划并执行行动的人工智能系统。基于大语言模型（LLM）的Agent将LLM作为"大脑"，结合规划、记忆和工具使用能力，实现自主完成任务的目标。

**Agent的核心公式**：
```
Agent = LLM（大模型）+ 规划（Planning）+ 记忆（Memory）+ 工具使用（Tool Use）
```

### 1.2 从“模式列表”到“体系架构”

传统的Agent设计指南常将各种模式平铺罗列，导致读者混淆“执行方式”、“启动时机”和“辅助功能”。本指南采用**四层架构**重新组织，帮助开发者系统性地理解与选用。

## 3. 触发方式详解

触发方式决定了一次Agent执行流程的**启动时机**与**启动来源**。同一套行动范式可通过不同触发方式激活。

| 触发类型 | 说明 | 典型场景 | 实现机制 |
|----------|------|----------|----------|
| **用户交互触发** | 用户在聊天界面主动发送消息 | 默认对话模式 | HTTP / WebSocket 消息 |
| **定时触发 (Cron)** | 按预设时间表周期性启动 | 每日报表、健康检查、定时提醒 | 调度器 + 持久化任务表 |
| **事件触发 (Webhook)** | 外部系统事件推送触发 | Git Push 后自动代码审查、订单创建后通知 | Webhook 监听 + 消息队列 |
| **API / 程序调用** | 其他服务或内部模块通过接口调用 | 被上游工作流引擎调度、UI按钮触发的后台任务 | REST / gRPC 接口 |
| **内部调度触发** | Agent内部一个任务完成后自动触发下一个 | Plan模式中子任务依赖链的自动续接 | DAG 引擎 / 状态机 |
| **条件触发** | 满足特定条件（如阈值超限）时自动启动 | 监控指标异常时触发诊断Agent | 规则引擎 + 指标采集 |

### 3.1 定时触发模式 (Cron) 详解

定时触发使Agent具备主动服务能力，其核心组件包括：

- **调度器**：基于 Cron 表达式或固定间隔触发。
- **任务持久化**：任务定义、执行历史、下次执行时间存入数据库。
- **执行上下文**：可绑定会话ID或全局用户ID，在触发时注入。

**创建方式**：
1. 用户通过客户端「定时任务」功能入口配置。
2. 对话中由大模型判断意图，调用「创建定时任务」工具写入库表。

**代码示例**（核心调度逻辑）：

```python
class CronScheduler:
    def __init__(self):
        self.jobs = {}
    
    def add_job(self, job: CronJob):
        self.jobs[job.id] = job
        self._calculate_next_run(job)
        self._reschedule()
    
    async def _execute_job(self, job: CronJob):
        # 执行绑定的Agent任务
        result = await agent.run(job.task_payload)
        # 更新状态并计算下次运行时间
        job.last_run_at = time.time()
        self._calculate_next_run(job)
```

## 4. 自动模式路由与守门机制

决策层负责根据任务特征，动态选择最合适的行动范式。它包含三个核心职责：

### 4.1 意图与场景识别

通过轻量级分类模型或LLM本身，对用户输入进行快速分析：

- **简单事实查询** → 路由至 Ask
- **需多步推理/工具调用** → 路由至 ReAct
- **需全局规划/多阶段** → 路由至 Plan
- **需多角色协作** → 路由至 Agent Team

### 4.2 显式模式选择与守门降级

即使用户在客户端**显式选择**了「计划执行」或「智能体团队」，系统仍须进行**守门检查**。若判定当前输入不适合该重型模式（如简单问候），则**实际执行降级为 Ask**，并在回复中向用户简要说明原因。

**守门规则示例**：
```python
def gateway_decision(user_input: str, selected_mode: str) -> str:
    if selected_mode in ["PLAN", "TEAM"]:
        if is_simple_query(user_input):   # 如“你好”、“今天星期几”
            return "ASK"   # 降级
    return selected_mode
```

### 4.3 路由器实现策略

| 策略 | 优点 | 缺点 | 适用阶段 |
|------|------|------|----------|
| **规则引擎** | 快速、可解释、零成本 | 覆盖有限，需维护 | 初期/MVP |
| **轻量分类模型** | 成本低、速度快 | 需训练数据 | 中等规模 |
| **LLM 动态判断** | 灵活、准确率高 | 成本较高、有延迟 | 复杂场景 |

## 6. 思考行动模式 (ReAct)

**定位**：通过“思考→行动→观察”循环迭代，边推理边执行。

**适用场景**：信息检索、多步工具调用、需实时反馈的动态任务。

### 6.1 ReWOO：免观察推理模式

**核心思想**：LLM一次性生成完整执行蓝图（计划），Worker按蓝图执行，中间观察结果不回传LLM，大幅降低Token消耗。

**对比**：
| 维度 | ReAct | ReWOO |
|------|-------|-------|
| Token消耗 | 高（每步回传） | 低（仅最终结果） |
| 灵活性 | 高 | 低 |

## 10. 行动范式编排与混合模式

### 10.1 范式组合原则

- **串联组合**：一个范式的输出作为下一个的输入（如 Plan → ReAct）。
- **嵌套组合**：一个范式内部节点采用另一范式（如 Workflow 节点内使用 ReAct）。
- **并行组合**：不同子任务同时采用不同范式（如 Agent Team 中各 Agent 使用不同模式）。

### 10.2 典型组合范式与案例速查

| 业务场景 | 触发方式 | 核心行动范式 | 辅助能力 |
|----------|----------|--------------|----------|
| 代码编写/调试 | 用户对话 | ReAct + Reflection | 上下文压缩 |
| 每日定时报告 | Cron | Plan / Workflow | State Tracking, HITL |
| 市场调研/深度研究 | 用户对话 | Plan + Agent Team | State Tracking |
| 智能客服 | 用户对话 | Ask + ReAct 路由 | RAG Memory |
| 自动化运维诊断 | 事件触发 | Workflow + ReAct | HITL, State |
| 内容审核 | API 调用 | Workflow (并行) | HITL |

### 10.3 场景案例详解

#### 场景一：代码编写助手
- **范式组合**：ReAct + Reflection
- **流程**：生成代码 → 执行测试 → 若失败则反思错误栈 → 修正代码 → 循环直至通过。

#### 场景二：每日定时报告
- **范式组合**：Cron触发 + Plan-and-Execute
- **流程**：每天9:00启动Plan → 查数据 → 分析趋势 → 生成图表 → 撰写邮件 → 经HITL审批后发送。

#### 场景三：多步骤市场调研
- **范式组合**：Plan + Agent Team
- **流程**：Plan拆解任务 → Search Agent搜索 → Analyst Agent分析 → Writer Agent汇总报告。

#### 场景四：智能客服
- **范式组合**：Ask（为主）+ ReAct（兜底）+ 路由决策
- **流程**：意图识别为简单查询则走Ask+RAG；需查订单等则走ReAct调用API。

#### 场景五：自动化运维诊断
- **范式组合**：Webhook触发 + Workflow + ReAct
- **流程**：监控告警触发Workflow标准检查 → 若无法定位则启动ReAct探索诊断 → 高危操作HITL确认。

#### 场景六：内容审核流水线
- **范式组合**：API触发 + Workflow并行节点
- **流程**：并行执行敏感词、LLM语义、图像审核 → 聚合打分 → 不确定内容转人工。

## 11. 状态追踪模式 (State Tracking)

**定位**：横切能力，通过异步状态表和小模型分析，为长时任务提供可观测性和进度管理。

**核心组件**：状态表、小模型分析器、异步追踪器。

**规范形态**：
- 嵌入 Ask / ReAct / Plan / Team 等路径，由模型门控是否发起状态追踪。
- 通过卡片展示进度，支持用户修订确认。
- 右侧分屏子页展示状态详情。

## 12. 人在环模式 (Human-in-the-Loop)

**定位**：横切能力，在关键节点暂停执行，等待人类确认、选择或修改。

**规范形态**：
- **非独立聊天模式**，而是嵌入执行路径的断点。
- 由模型判断是否需要人工介入（高风险、歧义、需授权）。
- 通过卡片消息请求确认，同一会话内续跑。

**典型场景**：删除确认、多方案选择、审批流。

## 13. 记忆管理与上下文压缩

### 13.1 记忆分层架构

- **短期记忆 (STM)**：当前会话上下文。
- **情节记忆 (Episodic)**：跨会话的任务历史轨迹。
- **长期记忆 (LTM)**：向量库中的知识、用户画像。

### 13.2 上下文压缩方案

当单次会话Token接近上限时，需压缩上下文。主流方案对比：

| 方案 | 原理 | 压缩率 | 适用场景 |
|------|------|--------|----------|
| LLM摘要 | 调用LLM生成结构化摘要 | 70%-90% | 长对话、多轮任务 |
| 直接截断 | 保留最近N条消息 | - | 简单对话 |
| 文件外存 | 大工具结果存文件，上下文仅保留路径 | - | 代码生成 |
| Token剪枝 | 小模型删除低信息量Token | 2x-20x | RAG、长文档 |
| 子Agent隔离 | 委派子Agent在隔离窗口执行 | - | 多阶段搜索 |

## 15. 多模态交互与Computer Use

**定位**：Agent通过“截图(Vision) + 模拟操作(Action)”控制计算机界面，适用于无API的遗留系统或网站自动化。
